# Mini Work Package: Stale `controllers-gl.js` Refresh Prompt For Run Pages
Status: Complete
Last Updated: 2026-02-13
Primary Areas: `wepppy/weppcloud/controllers_js/build_controllers_js.py`, `wepppy/weppcloud/controllers_js/templates/controllers.js.j2`, `wepppy/weppcloud/_context_processors.py`, `wepppy/weppcloud/static/js/controllers_gl_stale_check.js`, `wepppy/weppcloud/static/js/session_heartbeat.js`, `wepppy/weppcloud/templates/base_pure.htm`, `wepppy/weppcloud/routes/run_0/templates/runs0_pure.htm`, `wepppy/weppcloud/routes/fork_console/templates/rq-fork-console.htm`, `wepppy/weppcloud/routes/archive_dashboard/templates/rq-archive-dashboard.htm`, `wepppy/weppcloud/routes/readme_md/templates/readme_editor.htm`, `wepppy/weppcloud/templates/reports/_base_report.htm`, `wepppy/weppcloud/templates/reports/_page_container.htm`, `wepppy/weppcloud/controllers_js/__tests__/*`

See also: `wepppy/weppcloud/static/js/session_heartbeat.js` (existing "session expired" banner UX), `wepppy/weppcloud/utils/assets.py` (`ASSET_VERSION` cache busting)

## Objective
Add a UI/UX guardrail for run-scoped pages that depend on `controllers-gl.js` so that when the browser is running a stale bundle, the user is prompted to refresh the page to load the current client code.

In-scope pages (this package):
- run controls (`runs0_bp`): `/runs/<runid>/<config>/` (Pure run page)
- fork console: `/runs/<runid>/<config>/rq-fork-console`
- archive dashboard: `/runs/<runid>/<config>/rq-archive-dashboard`
- README editor: `/runs/<runid>/<config>/readme-editor`
- reports*: run-scoped report pages that load `controllers-gl.js` (Pure `_base_report.htm` and legacy `_page_container.htm`)

Out of scope:
- Other pages that happen to load `controllers-gl.js` (locations, batch runner, job dashboards, etc).

The prompt must match the look-and-feel of the existing "session expired" notification (bottom fixed warning banner with action buttons).

## Problem Summary
Some run-scoped templates still load `controllers-gl.js` without consistent cache busting (mix of `url_for('static', ...)` vs `static_url(...)`). When the bundle changes on disk/server but the browser reuses an older cached copy, UI behavior can silently diverge. The user should be told to reload, instead of encountering broken controls or confusing failures.

We already have a session-expiration banner (driven by `session_heartbeat.js`). This package adds the analogous "stale client bundle" banner for `controllers-gl.js`.

Note: This spec detects the "fresh HTML, stale JS" mismatch on page load. It does not proactively detect a long-lived tab that stays open across a deploy/rebuild until the user navigates/reloads. (Follow-up: add a periodic server check if we want that behavior.)

## Repo Reality (Serving + Versioning)
- `static_url()` appends `?v={{ ASSET_VERSION }}` via `wepppy/weppcloud/_context_processors.py`.
- `ASSET_VERSION` resolves from env/git/timestamp via `wepppy/weppcloud/utils/assets.py` and is set at app init (`wepppy/weppcloud/app.py`).
- `controllers-gl.js` includes a `Build date:` header and is generated by `wepppy/weppcloud/controllers_js/build_controllers_js.py` (built at container start via `docker/weppcloud-entrypoint.sh`).
- In production, Caddy serves `/weppcloud/static/*` from the mounted static dir; the bundle is also copied to `/srv/weppcloud/static/js/controllers-gl.js` via `CONTROLLERS_JS_EXTRA_OUTPUTS` (compose prod).

## Scope
- Define a stable "bundle identity" for `controllers-gl.js` that changes on every rebuild of the bundle.
- Provide the server's expected bundle identity to templates in scope.
- On the client, detect mismatch (or missing identity) and render a refresh banner.
- Ensure the banner styling/action layout matches `session_heartbeat.js` (copy exact inline style tokens).
- Wire the check into pages listed above (Pure templates + legacy report templates).
- Add regression coverage for the mismatch detection and banner behavior.

## Non-goals
- No broad rework of static cache headers, CDN behavior, or service worker policies.
- No forced auto-reload (avoid losing in-progress form edits); this is a prompt with explicit user action.
- No generalized "all bundles" stale detection (keep it limited to `controllers-gl.js`).
- No background polling for staleness in long-lived tabs in v1 (page-load check only).

## Contract: Bundle Identity + Expected Identity

### Canonical Identity Value
Use the bundle build timestamp already produced by `build_controllers_js.py`:
- `controllers_gl_build_id` (string): same format as the existing header comment build date, ex `2026-02-13T17:17:50Z`.

Rationale:
- Changes every time the bundle is rebuilt, even if git SHA / `ASSET_VERSION` does not change.
- Easy to embed in the bundle and easy for the server to read back from disk.

### Bundle Exposes Its Identity
`controllers-gl.js` must set a global early in the file:
- `window.__weppControllersGlBuildId = "<build_date>"`

Implementation: add a small prelude in `wepppy/weppcloud/controllers_js/templates/controllers.js.j2` that assigns the build id before any controller code executes.

### Server Exposes Expected Identity To Templates
Templates in scope must have access to:
- `controllers_gl_expected_build_id` (string)

Recommended injection:
- Add a lightweight helper that reads the first ~30 lines of `static/js/controllers-gl.js` and extracts the `Build date:` line.
- Cache by `(mtime, size)` so per-request overhead is effectively zero while still updating when the file changes.
- Make the value available via a Jinja context processor, so legacy templates (reports container) can use it too.

Recommended DOM surface:
- `<body data-controllers-gl-expected-build-id="{{ controllers_gl_expected_build_id|default('', true) }}">`

## UX Spec: "Refresh To Update" Banner

### Styling
Match `createSessionExpiredBanner()` in `wepppy/weppcloud/static/js/session_heartbeat.js`:
- fixed bottom bar, `z-index: 1200`
- background `#fff4e5`, top border `1px solid #f0b429`
- message color `#5f370e`, weight `600`
- action buttons styled with `#9f580a` border; primary filled `#9f580a` + white text, secondary white bg + dark text

### Copy
Message (short, action-oriented):
- "Update available. Reload to load the latest UI."

Actions:
- Primary: "Reload" (calls `window.location.reload()`).
- Secondary: "Dismiss" (optional). If implemented, it only hides the banner for the current page load.

### Behavior
- Require a non-empty expected build id:
  - If `expected_build_id` is missing/empty (including the literal string `"None"`), do not render the stale banner.
- Trigger when expected build id is present and either:
  - `window.__weppControllersGlBuildId` is missing, OR
  - `window.__weppControllersGlBuildId !== expected_build_id`.
- Do not trigger when the bundle is not loaded/required (avoid false positives by only running the check on templates that load `controllers-gl.js`).
- Banner must render once (idempotent by element id, ex `wc-stale-client-banner`).
- Banner stacking:
  - If `#wc-session-expired-banner` exists, set `staleBanner.style.bottom = sessionBanner.offsetHeight + "px"` to avoid overlap.
  - Listen for `wepp:session-heartbeat-expired` and recompute the bottom offset (session banner can appear after stale banner).
  - Recompute on `resize` (banner heights are responsive).

## Implementation Plan

### Phase 0: Bundle Publishes Build Id
- Update `wepppy/weppcloud/controllers_js/templates/controllers.js.j2`:
  - Add an early assignment for `window.__weppControllersGlBuildId`.
  - Keep it robust under non-browser execution (`typeof window !== "undefined"` guard).
- Rebuild bundle via `wepppy/weppcloud/controllers_js/build_controllers_js.py` and verify `controllers-gl.js` contains the global.

### Phase 1: Server Provides Expected Build Id
- Add `controllers_gl_expected_build_id` to the template context (new helper in `wepppy/weppcloud/utils/...` + hook it in `wepppy/weppcloud/_context_processors.py`).
- Extraction rules:
  - Read only the top portion of `static/js/controllers-gl.js`.
  - Parse the exact header line prefix `* Build date:` (fail closed: return `None` if not found).
- Template surface:
  - Add `data-controllers-gl-expected-build-id="{{ controllers_gl_expected_build_id|default('', true) }}"` to `<body>` on templates in scope.

### Phase 2: Client Check + Banner
Add a small standalone script (recommended new file):
- `wepppy/weppcloud/static/js/controllers_gl_stale_check.js`

Responsibilities:
- Read expected id from `document.body.dataset.controllersGlExpectedBuildId` (backed by `data-controllers-gl-expected-build-id`).
- Read actual id from `window.__weppControllersGlBuildId`.
- If stale: render the banner (copy styles from `session_heartbeat.js`).

Load order:
- Include the stale-check script after `controllers-gl.js` on each page in scope.
- Use `static_url('js/controllers_gl_stale_check.js')` (cache-busted by `ASSET_VERSION`) to reduce "stale checker" issues.
- Load both `controllers-gl.js` and `controllers_gl_stale_check.js` with `defer` (preserves execution ordering).
- The stale-check script must tolerate `document.body` being null (wait until DOM is ready before reading datasets).

### Phase 3: Wire Into Templates In Scope
Update script tags (minimum set):
- Prefer `static_url('js/controllers-gl.js')` for templates in scope that currently use `url_for('static', ...)` so query-param cache busting works consistently.
- Add the expected build id dataset field on those templates.
- Include `controllers_gl_stale_check.js` after `controllers-gl.js`.

Known templates to touch:
- `wepppy/weppcloud/routes/run_0/templates/runs0_pure.htm`
- `wepppy/weppcloud/routes/fork_console/templates/rq-fork-console.htm`
- `wepppy/weppcloud/routes/archive_dashboard/templates/rq-archive-dashboard.htm`
- `wepppy/weppcloud/routes/readme_md/templates/readme_editor.htm`
- `wepppy/weppcloud/templates/reports/_base_report.htm`
- `wepppy/weppcloud/templates/reports/_page_container.htm` (legacy)

### Phase 4: Regression Coverage
Add Jest coverage (jsdom) similar to the existing `session_heartbeat` test:
- New test module under `wepppy/weppcloud/controllers_js/__tests__/` that imports `../../static/js/controllers_gl_stale_check.js`.
- Tests:
  - renders no banner when build ids match
  - renders banner when build id missing
  - renders banner when build ids mismatch
  - renders no banner when expected build id is missing/empty
  - reload button emits `wepp:controllers-gl-stale-reload` (and attempts `location.reload` in browsers)
  - idempotency (no duplicate banners)

Add minimal Python coverage if server-side build-id parsing is non-trivial:
- Unit-test the parser against a small fixture header string (no filesystem dependency required).

## Manual Validation Checklist
1. Load a run page that uses `controllers-gl.js` (runs0, fork console, README editor, archive dashboard, reports).
2. Simulate mismatch:
   - rebuild `controllers-gl.js` (changes build id), but keep browser cached old copy (hard case), OR
   - use DevTools Local Overrides to change the served `controllers-gl.js` build id before reloading the page.
3. Confirm the stale banner appears and matches the session-expired banner styling.
4. Click "Reload" and confirm the banner disappears after reload (ids now match).

## Acceptance Criteria
- When `controllers-gl.js` is stale on any in-scope page, a bottom-fixed warning banner appears prompting a reload.
- Banner styling and layout matches the existing session-expired banner.
- Reload action works; no forced auto-reload occurs.
- No banner appears when bundle ids match.
- Regression tests cover the mismatch/missing-id cases.
