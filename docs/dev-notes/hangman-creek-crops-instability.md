# Hangman Creek Crops -- WEPP ifx Build Instability

## Summary

20 out of 5,426 hillslopes in the Hangman Creek watershed hang indefinitely when run with the **ifx-compiled** version of WEPP (`wepp_250915`) but complete successfully with the **gfortran-compiled** build (`wepp_dcc52a6`). The root cause is a numeric convergence failure in the erosion routing loop triggered by a specific combination of soil hydraulic parameters and management rotation under the ifx floating-point model.

**Workaround confirmed:** Flipping `ksflag` from 1 to 0 in the `.sol` files allows the ifx build to complete successfully.

**Status:** Unresolved upstream. Workaround identified but no fix has been applied to the ifx build or `wepp_runner`.

### Error Log Patterns

The 20 err logs split into three stall patterns:

1. **13 hillslopes** stall while "routing runoff & erosion event" (winter dates in Dec/Jan)
2. **5 hillslopes** stop right after the harvest record for day 288 ("ANNUAL CROP HARVEST DATE 288")
3. **2 hillslopes** freeze at "SIMULATION YEAR = 7"

In WEPP, these messages are emitted immediately before `route()` is entered (`wepp-forest/src/contin.for:1180`), so the hang occurs **inside the erosion routing loop**.

## Observations

1. **Identical management rotation:** All 20 failing hillslopes use the same two-year corn/soybean rotation in their 98.4 management files -- five tillage/planting passes per season repeated for 17 years. Only the Initial Condition block values differ slightly (`wepp/runs/p962.man:1`, `wepp/runs/p3131.man:91`).

2. **Hang location:** The hang occurs inside the erosion routing loop, entered via `route()` at `wepp-forest/src/contin.for:1180`.

3. **Soil file parameters:** Every soil file is version 9002 with:
   - `ksflag=1`
   - `ksatadj=0`
   - `ksatfac=0`
   - `ksatrec=0`
   - Imposed initial saturation of **0.75**
   - Files: `wepp/runs/p962.sol:92`, `wepp/runs/p1181.sol:92`

   The ifx build is therefore always running with internal hydraulic adjustments on a wet profile.

4. **Hydraulic conductivity split:**
   - **32.4 mm/h** -- 17 hillslopes (typical)
   - **9.72 mm/h** -- Naff series soils (`wepp/runs/p962.sol:95`, `wepp/runs/p534.sol:95`)
   - The low-ksat Naff soils account for the hung runs occurring during **January routing events**

5. **Uniform soil source:** All soils are silt-loam agricultural components generated by the same SSURGO pipeline on 2025-10-03. Parameter set across the failing population:
   - `salb`: 0.16--0.23
   - `ki`: ~4.8--5.2 x 10^4
   - `kr`: 0.0085--0.0136

6. **Management scheduling:** The rotation keeps residue option `resmgt=6` and applies five tillage events clustered around days 115--305 (`wepp/runs/p962.man:108` onward). The post-harvest to "Annual crop harvest" log suggests a scheduling mis-order can leave the solver waiting on residue until it times out.

7. **Initial condition variance:** Initial conditions vary from extremely loose (bulk density 0.95) to compact (1.25) across the same template (`wepp/runs/p3131.man:91`, `wepp/runs/p1284.man:91`). With low ksat, these transitions can drive shear spikes the new executable fails to resolve.

## Possible Causes

1. **Wet start + low-conductivity layers:** Naff/Tilma soils leave the routing loop with almost no infiltration capacity. Under the ifx binary, the sediment/deposition solver may never satisfy its convergence tolerance, whereas gfortran slips through.

2. **ksflag=1 with zeroed recovery factors:** The conductivity-adjustment routine still executes even with all recovery factors zeroed. If the ifx build changes floating-point behavior (e.g., stricter IEEE compliance, different rounding), this loop can stall on saturated surfaces.

3. **Tillage scheduling conflict:** The rotation's `resmgt=6` with five tillage events clustered around days 115--305 can cause a scheduling mis-order after harvest, leaving the solver waiting on residue indefinitely.

4. **Initial condition shear spikes:** Bulk density transitions from 0.95 to 1.25 across the same template, combined with low ksat, can produce shear spikes that the ifx executable's floating-point model fails to resolve.

5. **Numeric, not structural:** Because the gfortran build completes the same inputs, the issue is almost certainly a **numeric convergence problem** (hydrology/erosion) rather than malformed management or soil files. The ifx compiler likely handles floating-point edge cases (denormals, rounding, optimization) differently from gfortran.

## `wepp_runner` Hardening Findings

The investigation also identified robustness issues in `wepp_runner` when hillslopes hang:

### Key Findings

- The repeated "Hillslope simulations still running after 30 seconds" log is emitted by the wait loop in `wepppy/nodb/wepp.py:1804` whenever no futures finish within the window. With 5k+ hillslopes this is expected while long runs execute -- it signals the `ThreadPoolExecutor` is still waiting, not that it is stuck.
- Each hillslope future ultimately runs `run_hillslope` in `wepp_runner/wepp_runner.py:338`, which streams WEPP stdout line-by-line and only reports success when it sees `WEPP COMPLETED HILLSLOPE SIMULATION SUCCESSFULLY`. If the WEPP executable stalls, the loop never exits and the future stays "pending."
- The `.err` files are written by that same loop; if a WEPP process is still running (or hung) the success line will be missing even though `htop` shows the executable.

### Hardening Recommendations

1. **Per-hillslope timeout:** Add an optional timeout forwarded from `run_hillslopes` into `run_hillslope`. Inside `run_hillslope`, write stdout directly to the `.err` file (`stdout=process.wait(timeout=...)`) so we can detect overruns, kill the WEPP process, append a "TIMED OUT" marker, and raise a dedicated exception.

2. **Submission metadata tracking:** Track `future_to_id` / `future_to_start` map in `run_hillslopes` so when the 30s warning fires we can log how many hillslopes remain and which have been running longest.

3. **Graceful failure handling:** When a future raises (timeout or other error), record the offending `wepp_id` to a retry list and keep draining the rest instead of canceling. Report failed IDs at the end so they can be requeued or investigated individually.

## Confirmed Workaround

**Flipping `ksflag` to 0** in the `.sol` files for the stuck hillslopes (e.g., `wepp/runs/p962.sol:92`) and rerunning with `wepp_250915` results in **successful completion**. This confirms the conductivity-adjustment routine is the culprit under ifx floating-point semantics.

## Recommended Next Steps

1. **Flip `ksflag` to 0** for stuck hillslopes (confirmed working) or patch the soil pipeline to emit `ksflag=0` when initial saturation >= 0.75 and ksat is low.

2. **Test alternative soil parameters** for Naff/Tilma soils: lower imposed saturation (0.75 -> 0.5) or nudge top-layer ksat upward by 20% to see if routing converges with `ksflag=1`.

3. **Teach `wepp_runner` to auto-retry with gfortran:** Until the ifx build is patched, detect this management/soil signature and automatically retry with the gfortran binary so the batch can finish while collecting cases for an upstream bug report.

4. **File upstream bug report** against the ifx-compiled WEPP build with the collected failing cases (dumfounded-patentee_p962.zip).

## Background: Management Version Compatibility

The email thread also documents a parallel issue with management file versions:

- Hangman Creek managements from Mariana are **version 2017.1**
- WEPP currently expects **version 98.4** format
- Codex implemented a converter in `wepppy/wepp/management/managements.py` ([commit](https://github.com/rogerlew/wepppy/commit/0e019b49361d515256bdec3a92b82e202d2234f5))
- Downgrading from 2017.1 to 98.4 breaks some treatments (herbicide, resurfacing fraction / ground cover)
- `wepp-forest` also needs patching to handle 2016.3+ format
- Anurag noted wepp-forest doesn't have updated codes specifically done for NRCS

## Parallel Execution Memory Issue

When running hillslopes in parallel with the full 50k-hillslope WEPP binary (`wepp_250915`), each process consumed **2--2.5 GB** of memory, causing the server to run out of memory and swap.

**Solution:** Compile a second binary with reduced limits (`wepp_250915_hill`) using `_hill.inc` files with limits divided by 1000. Anurag confirmed bare minimum is 1 hillslope and 19 OFEs for parallel hillslope runs. The full compilation is only needed for big watershed channel routing.
