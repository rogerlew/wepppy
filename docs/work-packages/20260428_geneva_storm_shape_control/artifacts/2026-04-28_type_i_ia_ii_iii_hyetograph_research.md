# Type I/IA/II/III Hyetograph Research and Pre-Implementation Specification

**Date**: 2026-04-28 21:18 UTC
**Status**: Pre-implementation specification; code implementation must not start until the ordinate table artifact and metadata are present.

## Decision

Geneva Type I, Type IA, Type II, and Type III storm shapes must be implemented from NRCS 24-hour cumulative mass curves, using a checked-in ordinate table generated from NRCS WinTR-20 output as the implementation source of truth.

Do not scrape or transcribe secondary internet tables directly into Rust constants. Secondary tables may be used only as sanity checks after the authoritative WinTR-20-derived artifact is created.

## Authoritative Sources

- NRCS Title 210, National Engineering Handbook, Part 630, Chapter 4, "Storm Rainfall Depth and Distribution" (August 2019): `https://directives.nrcs.usda.gov/sites/default/files2/1712930592/29495.pdf`
- NRCS H&H Tools & Models page: `https://www.nrcs.usda.gov/conservation-basics/water/nrcs-hh-tools-models`
- NRCS WinTR-20 user guide notice for version 3.20.0000: `https://directives.nrcs.usda.gov/sites/default/files2/1712941092/UG%20210-21-5%20ENG%20WinTR-20%20Version%203.20.0000%2C%20Project%20Formulation%20-%20Hydrology%20Computer%20Program%20and%20User%20Documentation.pdf`

Relevant findings:

- NEH Chapter 4 defines rainfall distribution as cumulative rainfall fraction through the storm, beginning at 0 and ending at 1.
- NEH Chapter 4 describes standard Type I, IA, II, and III distributions as 24-hour NRCS synthetic design distributions.
- NEH Chapter 4 says the rainfall distribution tables may be requested as WinTR-20 output for export to spreadsheets or other hydrologic models.
- NEH Chapter 4 says a 24-hour design rainfall distribution includes embedded shorter-duration maxima, and that rainfall distribution for any shorter duration may be extracted from it.
- NRCS cautions that legacy Type I/IA/II/III distributions can be inconsistent with NOAA Atlas 14 site-specific ratios. Geneva must present these shapes as explicit user assumptions, not as automatic regional recommendations.

## Implementation Algorithm

Inputs:

- `distribution_type`: one of `type_i`, `type_ia`, `type_ii`, `type_iii`.
- `duration_minutes`: selected Geneva event duration.
- `depth_mm`: selected event depth for that duration.
- `time_step_minutes`: output time-step.

Validation:

- `duration_minutes` must be finite and `0 < duration_minutes <= 1440`.
- `depth_mm` must be finite and `> 0`.
- `time_step_minutes` must be finite and `> 0`.
- Output timestep count must remain under the existing Geneva kernel cap.

Interpolation:

- Treat the 24-hour source table as a piecewise-linear cumulative mass curve `F(t)`.
- Evaluate `F(t)` by linear interpolation between source ordinates.
- Clamp only exact floating-point endpoint noise; do not smooth or reshape the source curve at runtime.

Duration extraction:

1. Convert event duration to hours: `d = duration_minutes / 60`.
2. If `d == 24`, use the full source curve.
3. If `d < 24`, find the source-window start `a` in `[0, 24 - d]` that maximizes `F(a + d) - F(a)`.
4. Candidate starts must include source table times and source table times minus `d`, clipped to `[0, 24 - d]`; this is enough for a piecewise-linear curve because extrema occur at segment breakpoints.
5. If multiple starts tie within tolerance, choose the candidate closest to `12 - d / 2`; if still tied, choose the earliest.
6. Let `r = F(a + d) - F(a)`. Fail if `r <= 0`.
7. Build the normalized event curve:
   - `event_fraction(t) = (F(a + t_hours) - F(a)) / r`
   - force first output point to `0.0`
   - force final output point to `1.0`
   - enforce monotonicity within floating tolerance
8. `cumulative_rainfall_mm = event_fraction(t) * depth_mm`.

Output time-vector rule:

- Always include `0` and exact `duration_minutes`.
- Intermediate output points are regular `time_step_minutes` multiples strictly inside the duration.
- If `time_step_minutes >= duration_minutes`, output only `[0, duration_minutes]`.
- If duration is not evenly divisible by the step, the final interval is shorter.

Output metadata:

- `source_distribution_type`
- `source_curve_duration_hours = 24`
- `extraction_start_hours`
- `extraction_end_hours`
- `extraction_ratio_to_24h = r`
- `event_depth_is_duration_depth = true`
- `source_table_sha256`

## Why This Is Not Full-Curve Compression

Geneva frequency-panel cells provide event depths for specific durations such as 5, 10, 15, 30, 60, 120, and 1440 minutes. For a 60-minute cell, the selected `depth_mm` is the 60-minute depth, not the 24-hour depth.

Compressing the full 24-hour Type II curve into 60 minutes would discard the NRCS nested-duration relationship. Instead, the implementation extracts the maximum embedded 60-minute portion from the 24-hour cumulative curve and normalizes it to the selected 60-minute depth.

For 1440-minute cells, the extracted window is the full 24-hour curve.

## Validation Requirements

Rust tests must cover:

- CSV/constant provenance checks.
- Monotonic source columns and endpoints.
- Interpolation at exact and between-table times.
- Type II embedded ratios from NEH Figure 4-31, at least:
  - 5 minutes: `0.114`
  - 10 minutes: `0.201`
  - 15 minutes: `0.270`
  - 30 minutes: `0.380`
  - 1 hour: `0.454`
  - 2 hours: `0.538`
  - 3 hours: `0.595`
  - 6 hours: `0.707`
  - 12 hours: `0.841`
  - 24 hours: `1.000`
- Type II embedded ratios must match those NEH Figure 4-31 values within absolute fraction tolerance `<= 0.003`; failures must trigger source-export/interpolation investigation before any tolerance relaxation.
- For each Type I/IA/II/III distribution:
  - 1440-minute output starts at 0 and ends at depth.
  - 60-minute output starts at 0 and ends at depth after embedded-window normalization.
  - outputs are monotonic.
  - at least one representative shape differs from Uniform.
- For each Type I/IA/II/III distribution, short-duration output must prove embedded-window extraction instead of full-curve compression by asserting:
  - `extraction_ratio_to_24h < 1`.
  - extraction window is not the full `[0, 24]` curve.
  - output matches normalized source-window calculation rather than `F(t * 24 / d)`.
- Time-vector tests cover exact endpoints, non-divisible timestep durations, and `time_step_minutes >= duration_minutes`.
- Unsupported IDs fail closed.

Python/WEPPpy tests must cover:

- Run-batch schema accepts all six IDs and rejects unsupported IDs.
- Non-1440-minute Type I/IA/II/III output is built from an extracted window, not full-curve compression.
- Storm summary artifacts include extraction metadata for Type I/IA/II/III.
- Report assumptions identify legacy NRCS storm shape as a user-selected assumption.
