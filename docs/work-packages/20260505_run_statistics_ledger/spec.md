# Run Statistics Ledger Specification

**Status**: Draft for implementation  
**Created**: 2026-05-05 20:26 UTC  
**Package**: [20260505_run_statistics_ledger](package.md)

## Problem Statement

The old interface statistics were not a reliable report of model usage. On wepp1, `runs_counter.json` was generated by `wepppy/weppcloud/_scripts/compile_dot_logs.py` from dot access logs and current run-directory contents. That path had four defects for usage reporting:

- It used `first_access > datetime(2024, 1, 1)`, so the displayed start date was a hard-coded exclusive cutoff, not the first available data date.
- It skipped projects once their run directories disappeared or TTL state was no longer active, so historical project and run evidence was lost after 90-day rolling TTL deletion.
- It counted WEPP hillslopes with `len(wepp/runs/*.slp)`, which is an input/output-file count and cannot count repeated hillslope executions in the same project.
- It counted WATAR with `len(ash/*ash.csv)`, which is a legacy artifact glob and not a completed WATAR run count.

The replacement system must separate active-project inventory from historical execution accounting.

## Terms

**Project** means one WEPPcloud run directory identified by `runid` and `config`. A project can be active, queued for deletion, deleted from the filesystem, or known only from historical logs.

**Active project count** means the count of projects that currently exist, are not marked deleted by TTL metadata, and have enough metadata to classify by config. Active project counts are allowed to decrease when TTL deletes projects.

**Historical execution count** means the count of completed model work events recorded in the statistics ledger. Historical execution counts do not decrease when a project is deleted.

**WEPP hillslope run** means one completed WEPP hillslope simulation for one hillslope. If a project reruns hillslopes, the repeated execution must be counted again. For single-storm batch mode, record both completed task count and distinct hillslope count because one hillslope can produce multiple storm tasks.

**WATAR ash run** means one completed WATAR ash model simulation for one runnable hillslope. Skipped unburned/no-load hillslopes are not counted as WATAR ash runs.

**Dot access log** means a per-run hidden file named `.<runid>` under a run root or legacy root. Dot files establish access history and can seed project metadata, but they do not prove repeated WEPP or WATAR execution counts.

## Non-Negotiable Requirements

- The primary execution ledger must be PostgreSQL-backed with transactional inserts and uniqueness constraints so concurrent writers cannot corrupt counts.
- `project_deleted` is an audit event only and must never decrement historical WEPP/WATAR totals.
- Runtime hooks must append only after successful model completion. Failed runs must not increment completed execution counts.
- Repeated successful runs inside one project must create additional execution events and increase totals.
- Backfill must be idempotent. Running it twice must not duplicate historical events.
- Backfill from dot files must not invent repeated execution counts that the dot files cannot prove.
- Aggregates must distinguish `observed_runtime`, `legacy_artifact_inferred`, and `unknown_preledger` data.
- Public summaries must not expose email address, IP address, owner id, or raw filesystem paths unless an existing public contract already exposes that data.

## Storage Decision

PostgreSQL is the source of truth for event history. Redis is optional and may be used only for caching/materialized summaries, not as the durable event ledger.

Rationale:

- PostgreSQL gives transactional inserts, unique constraints for idempotent backfill, and safe multi-writer behavior across processes/hosts.
- Redis durability depends on AOF/RDB policy and eviction settings, and it is better suited for cache/stream use than long-horizon audit ledgers in this stack.
- Flat-file appends can be made safe on one host but remain brittle across host/process boundaries and harder to query for endpoint rollups.

## Ledger Schema

The implementation should create a table similar to `run_stats_events` in PostgreSQL.

Required columns:

- `id`: bigserial primary key.
- `event_id`: text or uuid, unique, not null.
- `event_type`: text, not null.
- `source`: text, not null.
- `occurred_at`: timestamptz, not null.
- `recorded_at`: timestamptz, not null, default `now()`.
- `runid`: text, not null.
- `config`: text, nullable.
- `config_bucket`: text, not null.
- `counts_json`: jsonb, not null.
- `meta_json`: jsonb, nullable.
- `dedupe_key`: text, nullable, unique when not null.

Required indexes:

- `(occurred_at)`
- `(runid, occurred_at)`
- `(config_bucket, occurred_at)`
- `(event_type, occurred_at)`

## Event Contract

Every event object must include:

- `schema_version`: integer. The initial version is `1`.
- `event_id`: string. Runtime events should use a UUID or another collision-resistant id. Backfill events must use deterministic ids so reruns are idempotent.
- `event_type`: string enum: `project_seen`, `project_deleted`, `wepp_hillslopes_completed`, `watar_hillslopes_completed`.
- `source`: string enum: `runtime`, `dotfile_backfill`, `legacy_artifact_backfill`, `ttl_gc`.
- `occurred_at`: UTC ISO-8601 timestamp ending in `Z`.
- `recorded_at`: UTC ISO-8601 timestamp ending in `Z`.
- `runid`: string.
- `config`: string or null.
- `config_bucket`: string.
- `counts`: object.

Optional fields:

- `job_id`: RQ job id or null.
- `run_mode`: for example `continuous`, `single_storm`, `single_storm_batch`, or null.
- `run_attempt_key`: stable identifier for a runtime invocation when available.
- `dedupe_key`: deterministic key for idempotent backfill events.
- `quality`: object describing confidence, for example `{"kind": "runtime_observed"}` or `{"kind": "artifact_inferred_minimum"}`.

The ledger must not store `user`, `email`, `ip`, or `owner_id`.

### Counts Object

For `project_seen`:

- `project_count`: `1`.

For `project_deleted`:

- `project_count`: `1`.

For `wepp_hillslopes_completed`:

- `hillslope_run_count`: completed future/task count.
- `distinct_hillslope_count`: distinct hillslope ids represented by the event when known.
- `single_storm_task_count`: task count for single-storm batch mode, or `0`.

For `watar_hillslopes_completed`:

- `watar_hillslope_run_count`: completed runnable ash hillslope count.
- `distinct_hillslope_count`: distinct WATAR hillslope ids represented by the event when known.
- `skipped_hillslope_count`: skipped unburned/no-load hillslopes when known.

## Config Buckets

Rollups must preserve exact `config` values and also compute a coarse `config_bucket` for legacy compatibility:

- `rhem`: config contains `rhem` and does not contain `eu`.
- `eu`: config contains `eu`.
- `au`: config contains `au`.
- `reveg`: config contains `reveg`.
- `disturbed`: default bucket.

Exact config summaries are canonical. Bucket summaries exist only for continuity with old counters.

## Runtime Event Sources

### WEPP Hillslopes

Append `wepp_hillslopes_completed` from `wepppy/nodb/core/wepp_run_service.py` after every successful `WeppRunService.run_hillslopes()` invocation and after all futures complete. The current function already tracks `futures_n` and `count`. The event should use the completed `count`, not the number of `.slp` files.

For normal continuous runs, `hillslope_run_count` and `distinct_hillslope_count` are usually the same. For single-storm batch mode, `hillslope_run_count` should be the completed future count and `distinct_hillslope_count` should be the watershed hillslope count.

### WATAR Ash

Append `watar_hillslopes_completed` from `wepppy/nodb/mods/ash_transport/ash.py` after `Ash.run_ash()` completes ash model execution and post-processing successfully. The current function builds `args` for runnable ash hillslopes. The event should use `len(args)` for `watar_hillslope_run_count`, not files under `ash/`.

If implementation captures skipped hillslopes, record them as `skipped_hillslope_count`; skipped hillslopes must not increase `watar_hillslope_run_count`.

### Project Deletion

Append `project_deleted` from `wepppy/rq/project_rq_delete.py` before the run directory is removed by TTL GC or explicit delete. This event is historical audit metadata only. It does not subtract from historical execution totals. Active project counts must still be derived from current active inventory, not by subtracting deletion events.

## Backfill

Backfill has two different confidence levels.

### Dot-File Backfill

Dot files can backfill `project_seen` events with earliest and latest access timestamps, runid, and config when config is recoverable from the current run directory or legacy metadata. Dot files can also support active-project inventory and landing-map outputs.

Dot files must not create WEPP hillslope execution totals or WATAR execution totals. They do not prove whether a project ran once, ran many times, or was only accessed.

### Legacy Artifact Backfill

For run directories that still exist, backfill may create one deterministic `legacy_artifact_backfill` event for minimum observed WEPP hillslope outputs and one deterministic event for minimum observed WATAR outputs.

WEPP artifact inference may use `wepp/runs/*.slp` as a minimum observed hillslope count. The event quality must be `artifact_inferred_minimum`.

WATAR artifact inference must not use `ash/*ash.csv`. It must use a strict output pattern for per-hillslope ash outputs, such as `H<integer>_ash.csv` or a documented current-format equivalent. Files such as stats summaries ending in `ash.csv` must not be counted.

For deleted projects with only dot files, execution counts are `unknown_preledger`. Do not fabricate totals.

## Statistics Endpoint Inventory

The current public statistics endpoints are in `wepppy/weppcloud/routes/stats.py`:

- `/getloadavg`
- `/access-by-year`
- `/access-by-month`
- `/stats`
- `/stats/<key>`

Current behavior:

- `/getloadavg` reports host load average and is not part of the ledger-backed rollup migration.
- `/access-by-year` and `/access-by-month` read `/geodata/weppcloud_runs/access.csv`.
- `/stats` and `/stats/<key>` read `/geodata/weppcloud_runs/runs_counter.json`.

Migration requirement:

- Keep `/getloadavg` behavior unchanged unless a separate package changes host-load reporting.
- Keep endpoint paths and response shapes compatible in the first rollout.
- Change endpoint data sources to PostgreSQL-backed rollups.
- Continue writing compatibility `runs_counter.json` during transition if external consumers still read files directly.

## Derived Outputs

The implementation must produce a canonical rich summary and compatibility outputs.

Canonical outputs:

- `run_statistics_summary.json`: overall active project counts, historical execution counts, source-quality totals, and generation metadata.
- `run_statistics_by_config.csv`: exact config, config bucket, active project count, historical WEPP hillslope run count, historical WATAR hillslope run count, and source-quality breakdowns.
- `stats_events_backfill_report.json`: backfill counts, skipped/unknown counts, and warnings.

Compatibility outputs:

- `access.csv`: preserve existing access-log output contract until route consumers migrate.
- `runid-locations.json`: preserve existing landing-map output contract.
- `run_counts.csv`: preserve or explicitly supersede after consumer audit.
- `runs_counter.json`: keep legacy keys such as `projects`, `hillruns`, and `ash_hillruns` during migration, but generate them from the new rollup and document their source mix.

## Compatibility Plan

This package is additive. It must not rename or remove existing route keys in `/stats` during the first rollout. Existing consumers of `runs_counter.json` continue to work, but the implementation must document that:

- `projects` means active project count in the compatibility view.
- `hillruns` means historical WEPP hillslope execution count where runtime ledger data exists plus clearly labeled artifact-inferred minimums.
- `ash_hillruns` means historical WATAR ash execution count where runtime ledger data exists plus clearly labeled artifact-inferred minimums.

Raw ledger events are an internal operational artifact and must not be exposed through public routes in this package.

## Regression Plan

Add focused tests for these exact behaviors:

- Ledger writer inserts valid event rows transactionally and rejects duplicate `event_id` or `dedupe_key` entries.
- Backfill deterministic ids prevent duplicate events when the backfill command runs twice.
- `WeppRunService.run_hillslopes()` records completed `count` and a repeated invocation doubles aggregate hillslope runs for the same project.
- Single-storm batch records task count separately from distinct hillslope count.
- `Ash.run_ash()` records `len(args)` after success.
- WATAR legacy backfill counts `H12_ash.csv` but does not count `foo_ash_stats_per_year_ash.csv` or other summary files.
- TTL deletion removes a project from active project counts but preserves existing historical WEPP and WATAR events in rollups.
- Dot-file-only deleted projects appear as project metadata or unknown historical rows without fabricated execution totals.
- Existing `/stats`, `/stats/<key>`, `/access-by-year`, and `/access-by-month` route tests still pass with database-backed rollups and compatibility outputs.

## Deployment and Validation Plan

1. Run the new tests with targeted `wctl run-pytest` commands.
2. Apply the schema migration on a production-like database and verify table/index presence.
3. Run `compile_dot_logs` or the new maintenance command against a local fixture with active, deleted, repeated-run, and WATAR cases.
4. On a production-like host, run backfill in dry-run/report mode first and inspect `stats_events_backfill_report.json`.
5. Enable runtime appends and observe for at least 14 days.
6. Compare old compatibility counters with rich summaries and document source-quality differences before any UI statistics are restored.
